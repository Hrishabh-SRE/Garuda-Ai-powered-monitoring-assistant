# Garuda — Executive Summary

Garuda is an AI-assisted observability and incident reasoning system we are building on top of our existing Prometheus, Thanos, and Grafana stack. The problem it solves: when something breaks in production — a latency spike, a 5xx burst, a pod crash loop — our SREs spend most of their time correlating metrics, logs, recent deployments, and Terraform changes across dozens of dashboards and Git commits before they can even begin fixing it. Garuda automates that correlation and uses an LLM (Gemini on Vertex AI) to produce a ranked list of probable root causes in plain English, with confidence scores and suggested remediation steps.

The build is phased. **Phase 1 — Ingestion:** we tap into what we already run — no new scrape targets, no new DaemonSets — pulling metrics via Thanos Query, logs via a ConfigMap patch on the existing Fluent Bit, Kubernetes events via the existing event-exporter, and deployment and IaC change events from ArgoCD and Atlantis webhooks; everything flows into the existing observability Kafka cluster. **Phase 2 — Processing:** deterministic anomaly detection via Prometheus recording rules plus a lightweight ML layer, log-pattern mining that compresses millions of log lines into a few hundred recurring templates, and a correlation engine that scores candidate causes by time proximity and by service-graph topology from Linkerd. **Phase 3 — Intelligence:** the compact correlated context plus retrieved similar past incidents and runbook snippets are sent to an LLM, which returns a ranked root-cause hypothesis — the LLM never sees raw logs, only compressed templates, which keeps token cost bounded. **Phase 4 — Presentation:** a Grafana plugin and a standalone incident page show the timeline, the supporting evidence, and the AI reasoning; SREs grade each hypothesis and that feedback loop continuously improves the ranking.

Two engineering disciplines underpin the whole system: Garuda is strictly read-only on the existing stack — it cannot degrade Prometheus or Thanos — and it runs in shadow mode for the first 30 days so we measure accuracy before anyone is paged on its output. The end state is faster MTTR, fewer context switches during incidents, and a self-improving knowledge base of how our platform actually fails.
